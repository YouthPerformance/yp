## Critical path triage: how to resolve each TODO without overbuilding

Claude’s TODO list is basically the “launch blockers list.” The key is to convert each TODO into either:

* **(A) working MVP implementation**, or
* **(B) a feature flag that removes it from the production path**

Here’s the practical mapping:

### 1) Auth / onboarding (Supabase TODOs vs Clerk envs)

Your audit mentions Supabase TODOs *and* Clerk env vars. That’s a red flag: **pick one auth path and delete the other.**

**Recommendation for speed:** If web‑academy already expects Clerk env vars, standardize on Clerk for auth in the app, and use Convex for your user record/profile writes.

Acceptance criteria:

* Given a user signs up, when they complete onboarding, then a Convex mutation writes:

  * `userId` (auth provider id)
  * role (athlete/parent)
  * age/sport/etc
  * onboardingComplete=true

(That aligns with your “Convex as memory” model. )

### 2) Workout completion → Convex

This is a retention feature and a data integrity feature. MVP:

* When a workout ends, write `{ workoutId, userId, completedAt, duration }`
* If write fails: queue locally (or at minimum log + show non-blocking toast)

### 3) Parent flow TODOs

You have two choices:

* **Implement real parent flow** (adds scope), or
* **Remove it from production path** (recommended for speed)

MVP approach:

* If “Parent” is selected, route to a page that clearly states “Parent onboarding is in progress; continue as athlete for now” OR block parent signup behind invite code.
* Important: if you collect data from minors, you may eventually need parental consent. Don’t pretend parent flow exists if it doesn’t.

### 4) Stripe checkout TODO

Do not build custom billing UI. MVP:

* Server route creates a Stripe Checkout Session (or uses a Checkout Link)
* client redirects to Stripe hosted checkout
* webhook (or return URL) updates entitlement in Convex

This matches your tech stack assumptions (Stripe in stack; MCP later; library wrapper in `yp-alpha` now is fine).
